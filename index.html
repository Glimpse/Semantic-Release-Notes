Semantic Release Notes

Semantic Release Notes (SRN) is a text-to-object-to-html conversion tool for application authors. SRN specificly aims to produce typed Release Notes from a Markdown document. This allows authors to write using an easy-to-read, easy-to-write plain text document, convert it to structured object (typically JSON or XML) and from this object, convert it into any styled format (typically html) or to consume the typed information within 3rd party applications.

Thus, SRN is two things:

  1. a plain text formatting syntax; and
  2. a software tool that converts plain text formatting to JSON/XML and HTML.
See the Syntax section for details pertaining to SRN fomatting syntax. You can also try it out, right now, using the online Editor or extending the provided examples.

Goals

The overriding design goals for SRN's formatting syntax is:

  • to make it as readable as possible,
  • to make this document parsable by a standard Markdown parser, and
  • to extract typed release information from the document.
The idea is that a SRN-formatted document should be publishable as-is, as plain text, without looking like it's been marked up with tags or formatting instructions. This works well in a Release Notes senario where plan text documents usually accompany the build. Additional, with the assumtptions around the formating of the Release document, SRN can extract semantic meaning from the plain text provided (i.e. the bugs fixed, features, included, release summaries, etc).

Influences

The estabilishment of the overall snytax and simplicity of the document is based on Markdown. As stated, a document based on SRN's syntax should be able to be parsed by a standard Markdown parser. This means that besides making the document human readable, it means that systems that aren't SRN aware but are Markdown aware, can still parse the documents.

The idea of being able to extract typed meaning from the document, isn't estabilished here, rather influcences where drawn from the typed format of TODO.txt files. Documents that follow the conventions of the TODO.txt "standard" are able to extract typed information within the TODO domain from the document. The origins on the stanrd are similar to those of Markdown (a need for human readability), but also a need to be able to define tasks, which are complete, what priority a given task has, etc.

Syntax

The axes of an Semantic Release Notes

Using the Markdown syntax in a SRN, you can create a list that's sliceable by various key axes.

  • Summary: At its most basic level, SRN should be able to take a paragraph and interprete that as the summary of what the release is about.
  • Item: A release note should be able to show which "items" are included in a given release. This is indicated via a standard Markdown ordered or unorderded list.
  • Section: Lets the author group items which represent logical sections within the application. These can be abratary in nature and specifc to the release notes which application is for. This is indicated via a standard Markdown header.
  • Priority: The release note should be able to let you define the importance feature or item that has been defined. This is indicated via a standard Markdown ordered list.
  • Category: Additionally the release note should be able to indicate what category an item is. For instance, "+New", "+Change", "+Fix", +Developer". This is indicated via a single word at the end of the list item that has a "+" prefix.
  • Release: As SRN allows for many releases to be defined within the one block of text, the system needs to provide a means by which an indovidual release can be identified. This is indicated via a standard Markdown header.
  • Project: As SRN allows for many projects with many releases to be defined within the one block of text, the system needs to provide a means by which an indovidual release can be identified. This is indicated via a standard Markdown header.

Summaries

It is useful to be able to provide summaries various parts of the release. Sections, Releases and Projects allow for multi-line multi-paragraph summaries, whilst the summary for an Item is more restictive, only allowing a single paragraph descriptions.

In each content block, a subset of Markdown is supported - _italic_, **bold**, `code` and [link](uri).

Examples: 
This is a _project_ summary with two paragraphs. 
Lorem ipsum dolor sit amet consectetuer **adipiscing** elit. 
Aliquam hendreritmi posuere lectus.

Vestibulum `enim wisi` viverra nec fringilla in laoreet
vitae risus. Donec sit amet nisl. Aliquam [semper](?) ipsum
sit amet velit.

                    
 
                
                    
Items

                    

                        When trying to extract indovidual items that make up a release, standard lists (ordered or unordered) are used. These 
                        lists indicate the items what are a part of the Section or Release they are a part of.
                    

                    

                        Since the syntax is a subset of Markdown the same list types are supported.   
                    

                    

                        Examples:
                        
 - This is the _first_ *list* item.
 - This is the **second** __list__ item.
 - This is the `third` list item.
 - This is the [forth](?) list item.

                        

                        

                    
 
                
                    
Sections

                    

                        When trying to extract indovidual items that make up a release, standard lists (ordered or unordered) are used. These 
                        lists indicate the items what are a part of the Section or Release they are a part of. These are usually used in larger 
                        applications when it's important to issolate out different sections of the application.
                    

                    

                        Sections are expressed through the use of headings. Also sections can have their own "Summaries" and "Items".
                    

                    

                        Examples:
                        
# Section
This is the summary for Section.
 - This is a Section scoped first list item.
 - This is a Section scoped second list item.

# Other Section
This is the summary for Other Section.
 - This is a Other Section scoped first list item.
 - This is a Other Section scoped second list item.
                        

                        

                        

                    
 
                 
                    
Priority

                    

                        When defining items, the priority of the fix can be expressed via a standard ordered list. Normally we expect 
                        ordered lists index to increment sequentially upwards. For our purposes, since the index represents teh the 
                        priority the values are always going to be 1, 2 or 3 - High, Normal and Minor. This means that when using 
                        priorities an index value is likely to be replicated. 
                    

                    

                        Since the syntax is a subset of Markdown the same ordered list types is supported. Note, the 
                        index order isn't important.
                    

                    

                        Example:
                        
 1. This is a High priority list item.
 1. This is a High priority list item. 
 2. This is a Normal priority list item. 
 1. This is a High priority list item. 
 2. This is a Normal priority list item.
 3. This is a Minor priority list item.
 3. This is a Minor priority list item. 
                        

                        

                        

                    
 

                    
Category

                    

                        When creating creating items it's useful to be able to assign a Category to an item. These Categories are 
                        fixed to the following types - New, Fix, Changed and Developer (where Developer describes a change that 
                        developers would be interested over users).
                    

                    

                        Markdown doesn't have an exact way of expressing this requirement, hence we use the following notation at 
                        the end of the item "+New".
                    

                    

                        Examples:
                        
 - This is a New list item. +New
 - This is a Fix list item. +Fix
 - This is a Change list item. +Change
 - This is a New list item. +New
 - This is a Developer list item. +Developer
                        

                        

                        

                    
 
            
                    
Release (Not Working)

                    

                        In some cases we want to have the one document that describes many releases. In this case, the 
                        syntax simply allows you to define a heading which is the Version Number. If you use this feature 
                        in conjunction with sections, Section headers are also altered.
                    

                    

                        For the purposes of interpretation of the version number, it is assumed that you are using 
                        Semantic Visioning - http://semver.org/. Normal Markdown headers 
                        are used to describe the version number for each release and the scope 
                        of reach release (which item, etc makes up the release).

Project (Not Working)

                    

                        In the same way that we may want to have a document that contains many Releases, we may want
